## 1. 스프링 입문 강의 요약



### 선택 키워드: 스프링 웹 개발 기초, 회원 관리 예제(백엔드 개발), 스프링 빈, AOP

### 스프링 웹 개발 기초
- 정적 컨텐츠: 서버에서 내용을 변경하거나 처리하지 않고 요청된 파일을 그대로 브라우저에 전달하는 방식
- MVC와 템플릿 엔진: 서버에서 프로그래밍을 통해 가공된 HTML을 전달하는 방식
- API 방식: 필요한 데이터 자체를  JSON과 같은 형식으로 클라이언트에 전달하는 방식



* 스프링을 활용한 웹 개발은 크게 세 가지 방식으로 구분된다. 각 방식은 데이터의 변형 여부와 전달 대상에 따라 차이가 있다.

## (1) 정적 컨텐츠
* 서버에서 별도의 처리 과정 없이 파일을 있는 그대로 웹 브라우저에 전달하는 방식
* 원리: 
1. 웹 브라우저의 요청을 내장 톰캣 서버가 받는다.
2. 스프링 컨테이너는 먼저 컨트롤러가 있는지 확인 (컨트롤러가 정적 파일보다 우선순위를 가짐.)
3. 일치하는 컨트롤러가 없다면 'resources/static' 경로에서 해당 파일을 찾아서 반환.



## (2) MVC와 템플릿 엔진 (Model, View, Controller)
* 서버에서 HTML을 동적으로 프로그래밍하여 렌더링 한 뒤 클라이언트에게 전달하는 방식. '관심사의 분리'를 위해 MVC 패턴 사용
- Model: 화면에 필요한 데이터를 담아 전달.
- View: 화면을 그리는 데만 집중.
- Controller: 비즈니스 로직을 처리하고 내부적인 움직임을 제어.
* 원리:
1. 컨트롤러가 로직을 처리한 후 뷰 이름(리턴값)과 데이터를 넘긴다.
2. 'ViewReslover'가 해당 뷰를 찾아 템플릿 엔진(Thymeleaf 등)에 연결.
3. 템플릿 엔진이 데이터를 기반으로 변환한 HTML을 웹 브라우저에 전달.



## (3) API
* HTML 화면을 보내는 대신, JSON과 같은 데이터 구조 포맷으로 데이터를 직접 전달하는 방식.
- ResponseBody: HTTP의 Body에 데이터를 직접 넣겠다는 의미. 뷰 리졸버 대신 HTTPMessageConverter가 작동함.
- JSON: Key와 Value로 이루어진 구조. 가장 표쥰족으로 사용되는 데이터 포맷.
* 원리:
1. @ResponseBody가 붙어있으면 객체를 반환할 때 JsonConverter가 동작하여 객체를 JSON으로 변환.
2. 변환된 JSON 데이터를 웹 브라우저나 클라이언트에 직접 반환한다.


* 내장 톰캣(Embedded Tomcat):** 별도의 웹 서버(WAS) 설치 없이, 애플리케이션 실행만으로 웹 서버가 내장되어 함께 구동되는 편리한 방식.




### 회원 관리 예제(백엔드 개발)
## 비즈니스 요구사항 정리
- 비즈니스 요구사항 정리
- 데이터: 회원ID, 이름
- 기능: 회원 등록, 조회
- (가상의 시나리오)


## 일반적인 웹 애플리케이션 계층 구조
- 컨트롤러: 웹 MVC의 컨트롤러 역할
- 서비스: 핵심 비즈니스 로직들 구현
- 도메인: 데이터베이스에 저장되고 관리되는 비즈니스 도메인 객체
- 리포지토리: 데이터베이스에 접근, 도메인 객체를 DB에 저장하고 관리


## 회원 도메인과 리포지토리 만들기
- 회원 도메인: 회원 정보를 담는 기본적인 객체로 id(Long)와 사용자가 입력하는 name(String)으로 구성
- 회원 리포지토리 인터페이스: 회원을 저장하고 찾는 기능이 필요하다는 설계도 역할
- 메서드: save, findById, findByName, findAll 기능 정의


## 회원 리포지토리 테스트 케이스 작성
* JUnit 테스트 프레임워크 사용 이유: main 메서드나 컨트롤러를 통해 직접 실행하는 방식은 준비과정 번거로움.
- 반복 실행 및 대규모 테스트를 한꺼번에 수행하기 어려움.
- Junit 장점: 코드를 코드로 검증할 수 있어 빠르고 안정적인 개발이 가능


## 테스트 코드의 독립성 보장
- 테스트 실행 순서가 보장되지 않음. 만약 이전 테스트에서 저장한 데이터가 메모리에 남아있다면 다음 테스트가 데이터 중복 등으로 실패할 수 있음.
- 방법: 각 테스트가 끝날 때마다 호출되는 @AfterEach 콜백 메서드를 사용하여 공용 데이터를 비워줘야 함.


## 회원 서비스 개발
- optional 
- null을 직접 반화할 경우 발생할 수 있는 NullPointException을 방지하고, 값이 없는 상황을 명시적으로 처리


## 테스트 코드의 중요성
- 안정성 확보: 수동으로 일일이 서버를 띄워 확인하는 번거로움을 줄이고, 로직의 오류를 조기에 발견하여 시스템의 안정성을 높임.
- 유지보수 용의성: 향후 코드 변경이나 기능 확장 시 발생할 수 있는 부작용을 차단하여 과감한 리팩터리 가능
- 살아있는 문서: 테스트 코드를 통해 해당 기능의 요구사항과 기대 결과를 명확히 정의함으로써 효율 극대화.





### 스프링 빈과 의존관계


* 스프링 빈 등록 방법 2가지
1. 컴포넌트 스캔과 자동 의존 관계 설정
2. 자바 코드로 직접 스프링 빈 등록


* 스프링 빈: 스프링 컨테이너는 애플리케이션 실행 시 필요한 객체들을 생성하고 관리
* 의존관계: 객체들 사이의 연결 고리를 만들어주는 과정 ex) 맴버 컨트롤러와 맴버 서비스를 통해 회원가입, 데이터 조회할 수 있어야 함. (서로 의존)


## 컴포넌트 스캔, 자동 의존관계 설정
- @Component 스캔: 스프링이 실행될 때 @Component 애노테이션이 붙은 클래스를 찾아 자동으로 스프링 빈으로 등록하는 방식
- @Autowired: 생성자에 @Autowired 애노테이션을 사용하면, 스프링 컨테이너가 관리하는 해당 빈을 찾아 자동으로 주입
- 이를 통해 객체 간 의존관계를 스프링이 직접 연결해줌. (ex: Controller가 Service를 사용할 수 있게 연결)


## 싱글톤
- 스프링은 컨테이너에 빈을 등록할 때 싱글톤으로 등록. 애플리케이션 전체에서 해당 객체는 단 하나만 생성되어 공유
- 장점으로 메모리 효율이 높고, 상태를 공유하지 않는 서비스나 리포지토리 객체를 관리하기에 최적화


## 자바 코드로 직접 스프링 빈 등록하기
- 컴포넌트 스캔 방식 대신, 별도의 설정 클래스를 만들어 스프링 빈을 직접 등록하는 방식
- 설정 클래스에 @Configuration을 붙이고, 각 메서드에 @Bean을 선언하여 반환된 객체를 스프링 컨테이너에 등록
- 직접 등록시 기존 서비스나 리포 클래스에 붙어있던 애노테이션 제거.


* 의존관계 주입 3가지 방법
1. 생성자 주입(의존관계가 실행 중 동정으로 변하는 경우는 거의 없으므로 권장)
2. 필드 주입
3. Setter 주입


* 직접 등록 방식을 사용하는 이유
- 컨트롤러, 서비스, 리포지토리처럼 정형화된 코드는 컴포넌트 스캔을 사용하는것이 편하다.
- 비정형화된 코드, 상황에 따라 구현 클래스를 변경해야 할 때는 직접 등록 방식 사용





### AOP (Aspect Oriented Programming)
- 공통 관심 사항과 핵심 관심 사항을 분리하여 프로그래밍 (공통 관심 사항 vs 핵심 관심 사항)


# AOP가 필요한 상황
- 예를들어 모든 메서드의 호출 시간을 측정해야하거나, 모든 로직에 로그를 남겨야 한다면?
- 시간 측정 로직은 핵심 비즈니스 로직이 아닌 공통 관심 사항이다. (한계...)
- 수백개의 메서드에 시간 측정 코드를 넣으면 유지보수 어렵고 복잡해진다.
- 비즈니스 로직과 공통 로직이 섞여 있어 가독성이 떨어짐.


- 시간 측정과 같은 공통 로직을 한 곳에 모으고(Aspect), 이를 원하는 곳에 적용 -> 코드 중복X, 핵심 로직 유지


# AOP 동작 방식
- AOP 적용 전: 컨트롤러가 실제 서비스(MemberService)를 직접 의존하여 호출한다.
- AOP 적용 후: 스프링은 실제 서비스 대신 가짜 서비스(Proxy, 프록시)를 앞 세운다.
1. 컨트롤러가 서비스 호출 -> 스프링 컨테이너가 만든 프록시 객체 먼저 실행.
2. 프록시에서 공통 로직 처리한 후, 실제 서비스 호출.
3. 이 방식을 통해 코드 수정 없이 원하는 시점에 공통 로직 삽입 가능.






### 2. SQL 기초: DDL과 DML

### 데이터베이스를 효과적으로 관리하기 위해 사용하는 SQL은 목적에 따라 여러 가지 언어로 구분.

### (1) DDL (Data Definition Language, 데이터 정의어)
- 데이터베이스의 객체(테이블, 인덱스, 뷰 등)를 생성하거나 변경, 그 구조를 정의, 수정, 삭제할 때 주로 사용함.
- 특징: 실행 즉시 데이터베이스에 반영되는 Auto Commit 특성을 갖는다. (취소가 어렵기 때문에 조심해야한다.)
* 코드 예시
  * CREATE 테이블 생성
      ```sql
        CREATE TABLE member (
            id BIGINT AUTO_INCREMENT PRIMARY KEY,
            name VARCHAR(255) NOT NULL
        );
        ```
      * ALTER: 테이블 구조 수정 (컬럼 추가 등)
        ```sql
        ALTER TABLE member ADD COLUMN age INT;
        ```
      * DROP: 테이블 자체를 완전히 삭제
        ```sql
        DROP TABLE member;
        ```
      * RENAME: 테이블의 이름을 변경
        ```
        RENAME TABLE member TO user_member;
        ```

### (2) DML (Data Manipulation Language, 데이터 조작어)
- 정의된 테이블 내부에 실제 데이터를 삽입, 조회, 수정 삭제할 때 사용
- 특징: 데이터의 실질적인 관리를 담당하며, 트랜잭션(Transaction) 내에서 작동한다. (Commit 이전에는 Rollback으로 복구가 가능)
* 코드예시
* SELECT 데이터 조회
* ```sql
      SELECT * FROM member WHERE name = 'Aim';
      ```
    * INSERT: 새로운 데이터 삽입
      ```sql
      INSERT INTO member (name) VALUES ('Hangjoon');
      ```
    * UPDATE: 기존 데이터 수정
      ```sql
      UPDATE member SET name = 'NewName' WHERE id = 1;
      ```
    * DELETE: 특정 행(Row) 데이터 삭제
      ```sql
      DELETE FROM member WHERE id = 1;
      ```
