# Chapter02. API 설계하기 - API명세서, RESTfulAPI

## 1. 학습 후기
이전에는 API라는 용어를 들어보기만 했을 뿐, 구체적으로 어떤 구조로 이루어져 있고 어떻게 설계해야 
하는지 막연한 상태였다. 그저 데이터를 주고받는 통로 정도로만 이해하고 있었다.

이번 워크북 학습을 통해 가장 크게 깨달은 점은 EndPoint(URI) 설계의 유연성이다. 
DB의 ERD 구조를 그대로 따라가는 것이 정답이라고 생각을 했는데, 
사용자의 목적지가 어디인지를 명확히 하는 것이 더 중요하다는 것을 배웠다.

단순히 주소를 만드는 것이 아니라, 명사(자원)는 URI에, 동사(행위)는 HTTP Method에 담아야 한다는 RESTful의 원칙을 직접 실습하며 익혔다. 
그치만 Request Body, Header, Query Parameter를 각각 어떤 상황에서 써야하는지 아직은 헷갈리는 것 같다.


## 2. 핵심 키워드 정리 (RESTful API, 멱등성이란?, HTTP 메서드 종류)
- RESTful API란?
API란 애플리케이션을 프로그래밍 할 때 보다 쉽게 할 수 있도록 해주는 도구들의 의미이다.

이해하기 쉽게 생각해보자면 식당에는 손님, 주방, 점원이 있다.
손님:client, 주방: server/DB, 점원: API 라고 생각할 수 있다.
점원(API)은 손님(client)에게 메뉴판을 주고, 주문을 받아 주방(server/DB)으로 전달을 하고, 완성된 음식을 다시 손님에게 전달한다.
API는 서로 다른 프로그램(앱과 서버)이 대화할 수 있게 해주는 ‘중간다리’, ‘규칙’ 이라고 볼 수 있다.

REST와 RESTful을 비유하자면,
REST: 요리 레시피
RESTful: 레시피를 아주 충실하게 잘 따라서 만든 요리

즉, RESTful API란 HTTP의 장점을 극대화하여, 자원(Resource)을 이름(URI)으로 구분하고, HTTP 메소드(GET, POST, PUT, PATCH, DELETE)를 통해 해당 자원의 상태를 주고 받는 설계 스타일을 준수한 API이다.

RESTful한 설계를 하기 위한 5가지 핵심 규칙

1. 자원(Resource)은 명사를 사용한다.
- /createReview처럼 동사를 포함하지 않는다. 행위는 HTTP 메서드가 대신해준다. 
2. 슬래시(/)는 계층 관계를 나타낸다.
- 컴퓨터 폴더 구조처럼 설계한다.
3. 마지막에 슬래시(/)를 붙이지 않는다.
- URL은 자원의 고유한 식별자이므로 마지막에 불필요한 슬래시는 넣지 않는다.
4. 하이픈(-)은 쓰되, 언더바(_)는 쓰지 않는다.
- 주소의 가독성을 높이기 위해 연결이 필요하다면 하이픈(-)을 사용한다.
5. 파일 확장자는 포함하지 않는다.
- 주소 끝에 .php, .json 같은 걸 붙이지 않는다. 어떤 기술로 만들었는지 숨기는게 보안상 안전


- 멱등성이란?
멱등성: 동일한 연산을 여러번 수행해도 결과가 달라지지 않는 성질을 말한다.

GET, PUT, DELETE는 멱등성 O
POST, PATCH는  멱등성 X

자동 복구 메커니즘, 서버 문제 시, 클라이언트가 같은 요청을 다시 해도 되는가의 판단 근거가 된다.



- HTTP 메서드 종류
HTTP 메서드: REST 방식으로  통신할 때 필요한 작업을 표시한 방법이다.
여러 메서드가 있지만 5개 메서드(GET, POST, PUT, PATCH, DELETE)는 CRUD(생성, 조회, 갱신, 삭제) 4가지에 대응된다.

GET - SELECT(Read조회) “서버에 있는 데이터를 가져와서 보여달라고 요청”
- 멱등성의 개념을 갖고 있어서 여러번 조회해도 리소스는 변하지 않는다.
- GET 요청에서 서버에 데이터를 전달하는 경우, 쿼리 파라미터를 통해 전달한다.
- 쿼리 파라미터는 주소창에 무방비로 노출되기 때문에 조심해야한다.
- GET 요청도 Body를 통해 데이터를 보낼 수 있지만, 많은 웹서버, 브라우저 등에서 GET은 주소만 읽으면 되는 줄 알고 Body를 무시한다. 
   → GET을 할 때는 Body를 쓰지말라는 암묵적인 약속.


---

POST - INSERT(create 생성) “새로운 데이터를 서버에 저장해달라고 요청”
- 데이터를 처리하고 메시지 Body를 통해 데이터를 전달한다.
- 단순히 데이터를 전달하는 것을 넘어, 이 데이터로 무언가를 만들어달라고 요청 가능.
- Body에 데이터를 담을 때 JSON 형식을 많이 사용한다.

---

PUT - UPDATE(update 수정) “기존 데이터를 수정해달라고 요청” (통째로)
- 쉽게  ‘파일 덮어쓰기’ 라고 생각할 수 있다.
- 리소스가 있으면 대체하고, 없으면 생성한다. 대체를 할 때는 전체 내용을 갱신한다.
- PUT은 멱등성이 있다. → ex) ‘이름: Aim’으로 바꿔달라고 PUT 요청을 하면 1000번을 보내도 그 데이터는 계속 ‘이름: Aim’ 이다.

---

PATCH - UPDATE(update 수정) “기존 데이터를 수정해달라고 요청” (부분 수정 가능)
- PUT에서 통째로 리소스를 수정한다면, PATCH는 부분 수정이 가능하다.
- PATCH 메서드를 지원하지 않는 서버가 있는데, 이 경우 POST를 사용한다.
  → POST는 새로운 데이터를 서버에 저장하는 용도이지만, 데이터를 처리해주는 넓은 의미를 갖고 있다.
- PATCH는 멱등성이 없다. “나이 칼럼(현재 20살)에 +10살 해줘.” → 1번 실행: 30살, 2번 실행: 40살 …

- 실무에서는 PUT보다 PATCH를 더 많이 사용한다. PUT을 사용하면 하나만 바꾸고 싶은데 모든걸 다시 보내는건 너무 비효율적이다.

---

DELETE - DELETE(Delete 삭제)
- 리소스를 제거할 때 사용한다.
- Hard Delete와 Self Delete를 구분하지 않고 클라이언트에게는 ‘삭제됨’이라는 결과만 주면 된다는 점은 백엔드 설계에서 중요한 Point이다.
- 리소스 삭제를 POST, PATCH로 명시하지 않도록 한다.
  → REST API의 가장 큰 규칙: 주소(URL)는 무엇(자원)인지 나타내고, 행동(Method)는 어떻게 할지 나타낸다.
- 나쁜 예시) 행동을 주소에 명시 POST/reviews/5/delete
- 좋은 예시) 행동을 메서드로 명시 DELETE/reviews/5


## 3. 미션
1. 인증(Auth)
사용자가 서비스를 이용하기 위한 가장 첫 단계이다.
- 회원가입(POST/auth/members): 사용자의 인적 사항과 선호 음식 카테고리를 입력받아 새로운 유저 자원을 생성.
- ERD상의 member와 member_prefer_category 테이블에 데이터가 저장된다.
- 로그인(POST/auth/login): 이메일과 비밀번호를 통해 본임인증, 성공시 API 호출에 필요한 AccessToken 발급

2. 홈 화면 및 지역 미션
사용자가 현재 위치한 지역의 미션을 탐색
- 지역별 미션 조회(GET/regions/{region}/mission): 특정 지역(region_Id)에 존재하는 모든 미션 목록을 가지고옴.

3. 리뷰관리
사용자의 경험을 기록하고 공유하는 기능
- 리뷰 작성(POST/stores/{store_Id}/reviews): 특정 가게에 대한 별점과 내용을 남긴다.

4. 미션 관리
사용자의 활동 상태를 관리하고 보상을 확인하는 과정
- 나의 미션 목록 조회 (GET/members/me/mission): 내가 도전 중인 미션과 이미 완료한 미션을 구분하여 조회한다.
- ?status= 쿼리 파라미터를 활용해 하나의 엔드포인트에서 상태별 필터링이 가능하도록 하였다.
- 미션 완료 처리(PATCH/members/{member_id}/me/member_missions/{mission_Id}): 미션 수행을 완료했을 때 상태를 업데이트한다.
- 기존 자원의 상태 일부를 수정하는 것이기 때문에 PATCH 사용함

5. 마에페이지
사용자의 개인 정보와 활동 요약을 확인한다.
- 내 정보 조회(GET/members/me): 닉네임, 포인트 조회, 리뷰 목록 조회 등 사용자의 종합적인 정보를 조회한다.
