## 스프링 웹 개발 기초
> 스프링 웹 개발의 3가지 방식
- 정적 컨텐츠: 서버에서 파일을 그대로 웹브라우저에 내려주는 방식
- MVC와 템플릿 엔진: 서버에서 프로그래밍을 통해 HTML을 동적으로 생성하여 내려주는 방식
- API 방식: 주로 JSON 데이터 포맷을 사용해 클라이언트나 서버 간에 데이터를 주고받는 방식

> 정적 컨텐츠란?
- 정적 컨텐츠는 서버가 파일을 그대로 전달하며, 별도의 프로그래밍 로직이 개입되지 않음
- 스프링부트는 정적 컨텐츠 기능을 자동으로 제공하며, 기본적으로 'static' 폴더에서 파일을 찾아 제공
- 정적 컨텐츠는 100% 동일한 파일이 반환되며, 서버에서 별도의 처리나 변형이 없음
- 먼저 컨트롤러에 해당 URL 매핑이 있는지 확인, 없으면 static 폴더에서 파일을 찾아 반환

> MVC와 템플릿 엔진이란?

### 개념과 필요성
- Model-View-Controller의 약자로, 소프트웨어 개발에서 관심사의 분리를 위해 도입된 구조
- 과거에는 컨트롤러와 뷰가 분리되지 않아, 뷰에 모든 로직이 집중되는 문제가 있었음
- 뷰는 화면 그리기에 집중하고, 컨트롤러와 모델은 비즈니스 로직과 내부 처리를 담당
- 관심사 분리를 통해 유지보수성과 역할 분담이 용이해짐

### 과거 개발 방식과 문제점
- 예전에는 JSP 파일 하나에 화면, DB 접근, 비즈니스 로직이 모두 포함되어 있었음
- 유지보수가 어렵고, 특정 개발자만이 코드를 이해할 수 있는 상황이 발생함
- MVC 패턴 도입으로 이러한 문제를 해결

###  MVC 패턴 적용
- 컨트롤러에서 파라미터를 받아 모델에 담아 뷰로 전달하는 구조를 사용함
- MVC 컨트롤러를 만들어, URL 파라미터를 받아 모델에 저장하고 뷰에 전달함
- 모델에 담긴 데이터는 템플릿 엔진을 통해 화면에 표시됨

### 템플릿 엔진과 Thymeleaf
- 템플릿 엔진은 HTML 파일을 서버에서 동적으로 변환해주는 역할을 함
- 타임리프의 장점은 HTML 파일을 서버 없이도 미리 볼 수 있다는 점임
- 실제 서버에서 동작할 때는 모델의 값으로 템플릿 내 변수가 치환됨

### 전체 동작 흐름
- 웹 브라우저가 요청을 보내면, 내장 톰캣 서버가 이를 받아 스프링에 전달
- 스프링은 컨트롤러를 호출하고, 모델 데이터를 뷰에 전달
- 뷰가 템플릿 파일을 찾아 템플릿 엔진에 넘기고, 엔진이 HTML로 렌더링하여 브라우저에 반환
- 정적 파일은 변환 없이 그대로 반환되지만, 템플릿 엔진을 거치면 동적으로 변환됨

> API란?
### 개념

-  HTML을 반환하는 대신, @ResponseBody를 사용하여 데이터를 직접 
HTTP 응답 Body에 담아 클라이언트에게 반환하는 방식

### @ResponseBody의 개념과 문자 반환
- 컨트롤러에 @ResponseBody를 사용하면 HTTP 통신 프로토콜의 응답 Body에 반환 데이터를 직접 넣어줌
- 템플릿 엔진과 달리 뷰를 거치지 않으며, 반환된 값이 뷰 없이 그대로 내려감
- 단순 문자열을 반환할 경우 HTML 태그 없이 순수한 문자 텍스트가 그대로 브라우저에 출력됨
- 하지만 실무에서는 단순 문자를 반환하는 방식을 거의 사용하지 않음

### 객체 반환과 JSON 포맷
- API 방식을 사용하는 주된 이유는 단순 문자가 아닌 '데이터' 자체를 넘겨주기 위함
- 스프링에서 객체를 생성하여 반환하면 기본적으로 JSON 포맷으로 변환되어 반환됨
- 과거에는 태그를 열고 닫아야 하는 무거운 XML 방식을 많이 썼으나, 
현재는 훨씬 심플한 JSON 방식으로 통일되어 널리 쓰이고 있음

### HttpMessageConverter의 원리
- 웹 브라우저의 요청이 오면 톰캣 내장 서버가 이를 스프링에 전달하고, 
스프링은 컨트롤러에 @ResponseBody가 있는지 확인함
- @ResponseBody가 있으면 기존의 뷰 대신 HttpMessageConverter가 동작함
- 반환값이 단순 문자일 경우 StringHttpMessageConverter가 동작하여 문자를 그대로 반환함
- 반환값이 객체일 경우 MappingJackson2HttpMessageConverter가 동작하여 객체를 JSON 스타일로 변환함
- 변환된 JSON 데이터는 HTTP 응답 바디에 실려 웹 브라우저나 안드로이드 등 요청한 클라이언트에게 최종 반환됨

## 회원 관리 예제 백엔드 개발

> 예제의 비즈니스 요구사항
- 회원의 아이디와 이름만을 관리하며, 회원 등록과 조회 두 가지 기능의 구현

> 웹의 기본 구조 
- 컨트롤러: 웹 MVC의 역할, API 생성 등 프론트엔드와의 인터페이스 담당
- 서비스: 핵심 비즈니스 로직이 구현되는 계층
- 도메인: 회원, 주문, 쿠폰 등 데이터베이스에 저장되는 객체
- 리포지토리: 도메인 객체를 저장하고 불러오는 역할

> 구조 설계
- 비즈니스 로직은 회원 서비스 객체에 구현
- 리포지토리는 인터페이스로 설계하여 구현체를 쉽게 교체할 수 있도록 함
- 초기 구현체는 메모리 기반 저장소이기에 향후 RDB, JPA 등 다양한 저장소 기술로 구현체를 교체할 수 있도록 설계함

> 회원 도메인(Member) 설계
- Member 클래스에는 시스템이 부여하는 id와 사용자가 입력하는 name 속성이 있음
- Getter와 Setter를 단순하게 모두 생성함

> 회원 리포지토리 인터페이스 설계 
- 리포지토리 패키지 생성 및 MemberRepository 인터페이스 정의
- 주요 기능: save(회원 저장), findById, findByName, findAll
- 반환 타입에 Optional 사용, null 반환 대신 Optional로 감싸 반환 (NullPointerException 방지, 데이터가 존재하지 않을 수 있음을 명시)

> 회원 리포지토리 구현
- 저장소로 HashMap<Long, Member> 사용, id 생성을 위한 sequence 변수 사용
- 실무에서는 동시성 문제로 ConcurrentHashMap(데이터 유실 방지), AtomicLong(원자성 보장) 등을 권장하지만 예제에서는 단순 구현
- 메서드
  - save: sequence 증가 후 id 세팅, store에 저장, 저장된 member 반환
  - findById: store에서 id로 조회, Optional로 감싸 반환
  - findByName: store의 values를 루프/람다로 돌며 name 비교, Optional로 반환
  - findAll: store의 values를 ArrayList로 반환

> 서비스 클래스 생성 및 구조
- 서비스는 회원 리포지토리와 도메인을 활용하여 실제 비즈니스 로직을 작성하는 역할
- 서비스를 만들기 위해 MemberRepository가 필요하며, 이를 의존성으로 가짐

> 회원 가입 로직 구현
- 회원 가입 메서드는 Member 객체를 받아 저장하고, 회원의 id만 반환하도록 설계
  - 회원 가입 시 같은 이름의 중복 회원은 허용하지 않는 비즈니스 규칙을 적용
- 중복 회원 검증을 위해 findByName 메서드를 사용하여 Optional로 반환된 결과를 ifPresent로 검사
  - 중복 회원이 존재할 경우 IllegalStateException을 발생
  - 중복 회원 검증 로직은 별도의 메서드로 추출하여 코드의 가독성과 재사용성을 높임

> 전체,단일 회원 조회
- 전체 회원 조회는 MemberRepository의 findAll을 호출하여 List를 반환
- 단일 회원 조회는 MemberRepository의 findById를 호출하여 Optional를 반환


> 테스트 케이스 작성의 필요성
- 클래스의 정상 동작을 검증하기 위해 테스트 케이스를 작성해야 함
- 기존확인 방식은 준비와 실행이 번거롭고 반복 실행에 어려움이 있음
- 여러 테스트를 한꺼번에 실행하기 어려움

> 테스트 메소드 작성 및 검증 방법
- 테스트 클래스 이름은 클래스명+Test 로 붙이는게 일반적
- @Test 어노테이션을 사용해 테스트 메소드를 작성함
- 회원 저장 기능 테스트: 멤버 객체를 저장하고, findById로 정상 저장 여부를 검증
- Optional에서 값을 꺼낼 때 get() 사용, 테스트 코드에서는 간단히 사용 가능

> 테스트의 독립성과 데이터 정리
- 테스트 실행 순서는 보장되지 않으므로, 테스트 간 데이터 의존성이 있으면 안 됨
- 각 테스트 종료 후 저장소를 비우는 clearStore 메소드와 @AfterEach 어노테이션 활용해서 독립성 확보

> 테스트 주도 개발(TDD)와 일반 테스트의 차이

### 일반 테스트

- 이미 작성된 코드가 잘 작동하는지 '검증'하는 데 초점
- 설계 ➔ 구현(운영 코드 작성) ➔ 테스트 코드 작성
- 아이디어나 설계를 빠르게 코드로 옮길 수 있어 초기 개발 속도가 빠름
- 테스트하기 어려운 구조로 개발되었을 경우 테스트 코드 작성을 포기하거나 누락하기 쉬움

### 테스트 주도 개발

- 테스트를 통해 시스템의 '설계'를 주도하고, 요구사항을 명확히 정의하는 데 초점
    - Red (실패하는 테스트 작성): 구현되지 않은 기능에 대한 테스트를 먼저 작성
    - Green (가장 빠른 통과): 테스트를 통과하기 위한 아주 단순하고 최소한의 코드를 작성
    - Refactor (리팩토링): 기능은 유지한 채 중복을 줄이고 코드 구조를 개선
- 사용자의 관점에서 코드를 먼저 생각하게 되므로, 자연스럽게 모듈화가 잘 된 유연한 코드가 작성
- 초기 학습 곡선이 가파르고, 코드를 작성하기 전 테스트를 먼저 고민해야 하므로 단기적인 개발 속도는 느리게 느껴질 수 있음

> 테스트 코드의 중요성
- 테스트 코드 없이 개발하는 것은 소규모 프로젝트에서는 가능하지만, 대규모 협업 및 코드베이스에서는 사실상 불가능에 가까움
- 빌드 툴과 연동해 테스트가 통과하지 않으면 다음 단계로 진행하지 못하게 할 수 있음

## 스프링 빈과 의존관계
> 스프링 빈과 의존 관계 설정
- 컨트롤러가 서비스를 통해 회원가입이나 조회를 수행하려면 두 객체 간에 의존 관계가 설정되어야 함
- 객체를 스프링 컨테이너에 등록하고 스프링으로 부터 받아 사용하도록 변경
- @Autowired를 지정하면 스프링 컨테이너가 뜰 때 연관된 스프링 빈을 찾아 자동으로 연결
- 스프링이 외부에서 객체 간의 의존 관계를 주입해 주는 것을 DI라고 함

> 컴포넌트 스캔 원리와 역할
- 클래스에 @Component 어노테이션이 있으면 스프링이 시작될 때 해당 객체를 생성하여 스프링 컨테이너에 자동으로 등록함
  - 어노테이션을 확인하여 스프링 빈으로 등록하는 과정을 '컴포넌트 스캔'이라고 함

> 컴포넌트 스캔 대상 및 범위
- 아무 패키지에나 @Component를 붙인다고 해서 무조건 스프링 빈으로 등록되는 것은 아님
- 기본적으로 메인 어플리케이션 클래스가 위치한 패키지와 그 하위 패키지들만 컴포넌트 스캔의 대상이 됨

> 스프링 빈 등록 특징(싱글톤)
- 싱글톤이란 해당 객체(컨트롤러, 서비스 등)를 컨테이너 내에 유일하게 딱 하나만 생성하여 등록하는 것을 의미
- 여러 곳에서 동일한 의존성을 요구할 때 똑같은 인스턴스를 공유하여 주입하므로 메모리가 절약됨

> 자바 코드로 직접 스프링 빈 등록
- 별도의 설정 클래스를 만들고 클래스 레벨에 @Configuration 어노테이션을 붙임
- 빈으로 등록할 객체를 생성해 반환하는 메서드를 작성하고 @Bean 어노테이션을 붙이면, 스프링이 시작될 때 이를 인식하여 스프링 빈으로 등록
- 의존 관계가 필요한 경우 서비스 빈을 생성하는 메서드 내부에서 리포지토리를 반환하는 메서드를 호출하여 주입
- 컨트롤러의 경우 어차피 스프링 컨테이너가 기본적으로 관리해야 하므로, 
기존대로 @Controller 어노테이션을 유지하고 컴포넌트 스캔과 @Autowired를 통한 의존성 주입 방식을 사용

> 의존성 주입(DI)의 3가지 방식

- 필드 주입: 클래스의 필드에 직접 @Autowired를 붙이는 방식으로, 중간에 의존성을 변경하거나 조작할 방법이 없어 권장하지 않음
- 수정자 주입: Setter 메서드에 @Autowired를 붙이는 방식으로, 의존성을 주입하기 위해 메서드가 Public하게 열려 있어야 하므로 누군가에 의해 의존성이 잘못 변경될 위험이 존재
- 생성자 주입: 생성자를 통해 의존성을 주입받는 방식으로, 애플리케이션 로딩 시점에 딱 한 번만 세팅되고 이후 외부에서 임의로 변경할 수 없도록 막을 수 있어 가장 권장되는 방식

> 컴포넌트 스캔과 직접 설정의 활용 기준
- 컴포넌트 스캔: 일반적으로 애플리케이션에서 작성하는 정형화된 역할인 컨트롤러, 서비스, 리포지토리 같은 코드에 주로 사용
- 자바 코드 직접 설정: 역할이 정형화되지 않았거나, 상황에 따라 내부 구현 클래스를 동적으로 변경해야 할 때 주로 사용
## AOP

>AOP의 필요성(예시)

- 시스템 내의 모든 메서드 호출(실행) 시간을 측정해서 남겨야 하는 상황을 가정 
- 기존 방식으로 해결하려면 수백, 수천 개의 메서드를 일일이 열어 
시작점과 끝점에 시간 측정 로직을 복사해서 붙여넣어야 하는 중노동이 발생
- 만약 시간 측정 단위를 초에서 밀리초(ms)로 변경해 달라는 등 요구사항이 바뀌면, 
앞서 수정했던 모든 메서드를 다시 찾아가 일일이 코드를 뜯어고쳐야 한다

> 기존 개발 방식의 한계와 문제점
- 시간 측정과 같은 부가적인 로직이 핵심 비즈니스 로직과 한데 뒤섞이게 되어 코드의 가독성이 떨어지고 유지보수가 여려워짐
- 메서드의 시작과 끝을 동시에 감싸야 하는 시간 측정 로직의 특성상, 
이를 단순히 별도의 공통 메서드로 깔끔하게 분리하기가 매우 까다로움

> '핵심 관심 사항' 에서 '공통 관심 사항'을 분리하는 기술 AOP
- 회원 가입이나 회원 조회와 같은 시스템의 본질적인 비즈니스 로직은 '핵심 관심 사항'이라고 부르는 반면 
여러 메서드와 클래스에 걸쳐 공통으로 적용되어야 하는 시간 측정, 로깅 같은 부가 기능 로직은 '공통 관심 사항' 이라고 한다
- 흩어져 있는 '공통 관심 사항' 코드를 '핵심 관심 사항'에서 깔끔하게 분리해 내는 기술이 바로 AOP이다

> AOP 구현 방법
- 스프링에서는 AOP를 쉽게 구현할 수 있도록 지원하며, @Aspect 어노테이션과 Advice 개념을 활용함
- JoinPoint 등을 사용하여 메서드 실행 전후에 공통 로직을 삽입할 수 있음
- @Component로 스프링 빈에 등록하여 관리하며, @Around 어노테이션으로 적용 대상을 지정함

> AOP 적용 및 효과
- 기존에 각 서비스에 흩어져 있던 시간 측정 코드를 제거하고, 핵심 로직만 남길 수 있음 
- 어디서 병목현상이 발생하는지 쉽게 파악할 수 있음 
- 적용 대상을 패키지, 클래스 등 다양한 조건으로 세밀하게 지정할 수 있음 (09:00)

> AOP의 동작 원리 (스프링 기준)
- 스프링 컨테이너가 실제 빈 대신 가짜 빈을 생성하여 주입
- 가짜 빈이 메서드 호출을 가로채고, 공통 로직을 실행한 후 실제 빈의 메서드를 호출
-프록시(가짜 빈) 방식 외에도 컴파일 타임에 코드를 삽입하는 방식 등 다양한 구현 방법이 존재
> AOP의 장점 및 결론
- 핵심 로직과 공통 로직의 분리로 코드의 가독성과 유지보수성이 향상됨
- 변경 사항이 생기면 공통 로직만 수정하면 되어 효율적임 
- 원하는 적용 대상을 유연하게 선택할 수 있음

# SQL 기초: DDL vs DML

---

## DDL(데이터 정의 언어, Data Definition Language) 이란?
### 정의

- 데이터베이스에서 스키마를 정의하거나 수정하고, 
테이블, 뷰, 인덱스 등의 데이터 객체를 생성하거나 변경하는 데 사용되는 SQL 명령어다.

### 특징

- 데이터베이스의 뼈대가 되는 구조를 설계하고 관리하는 데 핵심적인 역할을 합니다.
- 데이터베이스 관리 시스템 내의 메타데이터 구조를 실질적으로 정의합니다.

### 코드 예시
- CREATE: 데이터베이스나 테이블, 뷰, 인덱스 등을 새롭게 생성합니다.
``` 
*데이터베이스 생성
CREATE DATABASE my_database;

*테이블 생성
CREATE TABLE users (
    id INT PRIMARY KEY,
    name VARCHAR(50)
);
```
- ALTER: 기존에 존재하는 데이터베이스나 객체의 구조를 수정합니다.
```
*컬럼 추가
ALTER TABLE users ADD email VARCHAR(100);
```
- DROP: 데이터베이스나 객체를 완전히 삭제합니다. (해당 객체와 관련된 데이터가 모두 손실됩니다.)
```
*테이블 삭제
DROP TABLE users;
```
- TRUNCATE: 테이블의 구조는 그대로 남겨두고, 안에 들어있는 모든 데이터를 한 번에 삭제합니다.
```
*테이블 내 모든 데이터 삭제
TRUNCATE TABLE users;
```
- RENAME: 데이터베이스나 객체의 이름을 변경합니다.
```
*테이블 이름 변경
RENAME TABLE old_table_name TO new_table_name;
```
## DML(데이터 조작 언어, Data Manipulation Language) 이란?

### 정의

- 데이터베이스 내 테이블에 저장된 데이터를 실질적으로 조회, 삽입, 수정, 삭제하는 데 사용되는 SQL 명령어이다.

### 특징

- 사용자나 애플리케이션이 데이터베이스와 상호작용하며 실질적인 데이터를 관리할 때 사용된다.
- DDL을 통해 만들어진 테이블 구조 안에서 데이터 레코드를 다루는 역할을 합니다.

### 코드 예시

- SELECT: 테이블에서 조건에 맞는 데이터를 조회합니다.
```
*특정 데이터 조회
SELECT id, name FROM users WHERE id = 1;
```
- INSERT: 테이블에 새로운 데이터(행)를 삽입합니다.
```
*새로운 행 삽입
INSERT INTO users (id, name) VALUES (1, '홍길동');
```
- UPDATE: 테이블에 이미 존재하는 특정 데이터의 값을 수정합니다.
```
*특정 행의 값 수정
UPDATE users SET name = '김철수' WHERE id = 1;
```
- DELETE: 테이블에서 조건에 맞는 특정 데이터를 삭제합니다.
```
*조건에 맞는 행 삭제
DELETE FROM users WHERE id = 1;
```
## DDL과 DML의 차이점

- 작업 대상의 차이: DDL은 데이터베이스의 구조 및 객체 자체를 대상으로 작업하며, 
DML은 그 구조 안에 담긴 데이터레코드를 대상으로 작업합니다.

- 사용 목적의 차이: DDL은 데이터를 담을 그릇을 만들고 변형하는 건축적인 작업이라면, 
DML은 완성된 그릇 안에 내용물을 넣고 빼고 바꾸는 작업입니다.

- 실행 범위: DDL의 실행은 데이터베이스의 메타데이터 자체를 변경하지만, 
DML은 특정 테이블 내의 데이터 값과 개수만 변경합니다.
