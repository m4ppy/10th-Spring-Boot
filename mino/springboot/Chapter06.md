### JPA란?
**자바 애플리케이션에서 관계형 데이터베이스를 사용하는 방식을 정의한 ORM(Object-Relational Mapping) 기술 표준으로
자바 객체와 DB 테이블을 매핑하여 SQL 직접 작성 없이 객체 중심으로 데이터를 다루며, 생산성과 유지보수성을 크게 향상시킵니다. 대표적인 구현체로는 Hibernate가 있습니다.**

**JPA를 사용하는 이유**

DBC나 MyBatis 같은 SQL 매퍼 방식은 개발자가 직접 SQL을 작성하고 관리해야 했습니다. JPA는 다음과 같은 핵심적인 이유로 도입되었습니다.

- **객체 중심 개발**: 데이터베이스 테이블 중심에서 벗어나 자바 객체(Entity) 중심으로 비즈니스 로직을 설계할 수 있습니다.
- **패러다임 불일치 해결**: 상속, 연관관계, 객체 그래프 탐색 등 객체 지향 언어의 특징과 관계형 데이터베이스의 구조적 차이를 자동으로 매핑해 줍니다.
- **데이터 접근 추상화**: 특정 데이터베이스 문법(방언)에 종속되지 않아, DB를 교체하더라도 코드를 수정할 필요가 거의 없습니다.

**장점과 단점**

장점

- **생산성 향상**: 반복적인 CRUD SQL 작업을 간단한 메서드 호출로 처리합니다.
- **유지보수 용이**: 필드 변경 시 관련 SQL을 일일이 수정할 필요 없이 엔티티만 수정하면 됩니다.
- **데이터 일관성 보장**: 같은 트랜잭션 내에서 조회한 엔티티의 동일성을 보장하며, 1차 캐시를 통해 성능을 최적화합니다.
- **지연 로딩(Lazy Loading)**: 객체가 실제 사용되는 시점에 데이터를 조회하여 자원을 효율적으로 관리합니다.

단점

- **높은 러닝 커브**: 내부 동작 방식과 성능 최적화(N+1 문제 등)를 깊이 이해해야 의도치 않은 성능 저하를 막을 수 있습니다.
- **복잡한 쿼리의 한계**: 매우 복잡한 통계 쿼리나 대량의 배치 작업에는 네이티브 SQL이나 MyBatis보다 성능이 떨어질 수 있습니다.
- **의도치 않은 쿼리 발생**: 잘못된 설계 시 개발자가 의도하지 않은 수많은 쿼리가 발생하여 성능 병목이 생길 수 있습니다.

**사용 예시**

- **엔티티 정의 (Entity)**: 클래스에 `@Entity`와 `@Id` 어노테이션을 붙여 DB 테이블과 매핑합니다.
- **리포지토리 인터페이스 (Repository)**: `JpaRepository` 인터페이스를 상속받기만 하면 기본적인 저장, 수정, 삭제, 조회 기능을 별도 구현 없이 사용할 수 있습니다.
```
// 1. 엔티티 정의
@Entity
public class Member {
    @Id @GeneratedValue
    private Long id;
    private String name;
}

// 2. 리포지토리 인터페이스 정의 (구현체 없이 인터페이스만 작성)
public interface MemberRepository extends JpaRepository<Member, Long> {
    List<Member> findByName(String name);
}

// 3. 서비스 계층에서의 사용
@Service
public class MemberService {
    @Autowired 
    private MemberRepository memberRepository;
    public void join(Member member) {
        memberRepository.save(member);
    }
}
```
### N+1 문제란?
**ORM(JPA 등) 기술 사용 시, 1번의 쿼리로 N개의 데이터를 조회했으나 연관된 데이터를 가져오기 위해 N번의 추가 쿼리가 발생하여 성능이 저하되는 현상입니다.**

- **발생 배경:** JPA의 연관관계 매핑에서 EAGER(즉시 로딩) 또는 LAZY(지연 로딩) 전략을 사용할 때, 예상치 못한 시점에 추가 조회가 일어나면서 발생합니다.

**예시**
- '회원(Member)'이 10명(N) 있고, 각 회원이 '주문(Order)' 목록을 가지고 있는 경우.
- `memberRepository.findAll()`로 모든 회원 조회(1번 쿼리) 후, 각 회원의 주문 목록을 가져오기 위해 10번의 추가 쿼리가 발생

이러한 문제가 발생하는 이유는 JPA의 두 가지 특성이 주된 요인이다

- JPQL의 SQL중심적 처리방식 :  명령 처리시 필요한 엔티티를 한번에 가져오는 것이 아니라 하나씩 가져오는 문제
- Fetch 전략 문제 : 지연로딩의 가짜 프록시 생성과정에서 프록시 안은 비어있는 상태이기 때문에 접근 시 데이터가 없어 라고 인식해 쿼리를 날리게 되고
  즉시 로딩에서는 엔티티 조회시 데이터가 필요하다고 즉시 판단하기 때문에 연관된 데이터를 조회하기 위해 n번의 쿼리를 실행합니다.

해결방법

- **Fetch Join (가장 일반적):** SQL JOIN을 사용하여 한 번의 쿼리로 연관된 데이터까지 모두 가져오는 방식.
- **EntityGraph:** `@EntityGraph` 어노테이션을 사용하여 필요한 연관 관계를 Fetch Join 방식으로 가져옴.
- **Batch Size 설정:** `hibernate.default_batch_fetch_size`를 설정하여, 1번의 쿼리로 개의 연관 객체를 `IN` 절로 가져옴.
### 지연로딩과 즉시로딩의 차이는?
**1. 주요 차이점 비교**

| **구분** | **지연 로딩 (LAZY)** | **즉시 로딩 (EAGER)** |
| --- | --- | --- |
| **로딩 시점** | 연관된 엔티티를 **실제로 사용할 때** DB를 조회함 | 엔티티 조회 시 **연관된 엔티티도 함께** 조회함 |
| **조회 방식** | 프록시(Proxy) 객체를 생성하여 초기화 지연 | JOIN 쿼리를 사용하여 한 번에 조회함 |
| **장점** | 초기 로딩 시간 단축 및 메모리 사용 효율성 증대 | 연관된 데이터를 즉시 사용할 수 있어 편리함 |
| **단점** | 연관 데이터 접근 시 추가 쿼리로 성능 저하 가능 | 불필요한 데이터 로딩으로 인한 성능 저하 및 N+1 문제 |

**2. 기술적 상세 특징**

- **지연 로딩 (LAZY)**
    - **프록시 객체 활용:** 메인 엔티티 조회 시 연관 엔티티 자리에 가짜 객체인 **프록시**를 넣어두며, 해당 프록시의 메서드를 호출(예: `member.getTeam().getName()`)할 때 실제 DB 쿼리가 실행됩니다.
    - **주의 사항:** 영속성 컨텍스트(Session)가 종료된 후 프록시 객체에 접근하면 `LazyInitializationException`이 발생합니다.
    - **기본 전략:** `@OneToMany`, `@ManyToMany`와 같은 컬렉션 관계의 기본 Fetch 전략입니다.
- **즉시 로딩 (EAGER)**
    - **JOIN 쿼리 실행:** JPA 구현체는 대개 SQL의 `LEFT OUTER JOIN` 등을 사용해 연관된 데이터를 한 번의 쿼리로 가져옵니다.
    - **위험성:** 실무에서는 예상치 못한 SQL이 실행되거나 복잡한 연관 관계에서 **N+1 문제**를 일으키기 쉬워 가급적 사용을 지양하고 지연 로딩을 권장합니다.
    - **기본 전략:** `@ManyToOne`, `@OneToOne`과 같은 단일 객체 관계의 기본 Fetch 전략입니다.
### JPQL란?
JPA(Java Persistence API)에서 정의한 **엔티티 객체를 대상으로 하는 객체지향 쿼리 언어**입니다.

**1. 주요 특징 및 정의**

- **객체지향 쿼리:** 데이터베이스의 테이블을 대상으로 직접 쿼리하는 SQL과 달리, 자바 클래스인 **엔티티 객체와 그 필드**를 대상으로 조회합니다.
- **DB 독립성:** 특정 데이터베이스(MySQL, Oracle 등)의 고유 문법에 의존하지 않고, JPA가 각 DB 방언(Dialect)에 맞춰 적절한 SQL로 변환하여 실행합니다.
- **추상화된 SQL:** 문법 자체는 SQL과 유사하여 `SELECT`, `FROM`, `WHERE`, `GROUP BY`, `HAVING`, `JOIN` 등을 모두 지원합니다.

**2. SQL과의 핵심 차이점**

| **구분** | **SQL** | **JPQL** |
| --- | --- | --- |
| **대상** | 데이터베이스 **테이블** | 애플리케이션 **엔티티 객체** |
| **속성 지정** | 컬럼 명칭 사용 | 엔티티 클래스의 **필드(변수) 명칭** 사용 |
| **반환 값** | 로우(Row) 데이터 | 엔티티 객체 또는 DTO 객체 |

**3. 기술적 제약 및 주의사항**

- **컴파일 타임 체크 불가:** JPQL은 기본적으로 **문자열(String)** 기반으로 작성되기 때문에, 오타가 있어도 컴파일 시점에는 알 수 없고 애플리케이션 실행(Runtime) 중에 쿼리가 호출될 때 오류가 발견됩니다.
- **동적 쿼리 작성의 어려움:** 문자열을 더해가며 복잡한 동적 쿼리를 만드는 것이 번거로우며, 이를 보완하기 위해 **Criteria API**나 **QueryDSL** 같은 기술이 함께 사용되기도 합니다.
- **대소문자 구분:** JPQL 키워드(`SELECT`, `FROM` 등)는 대소문자를 구분하지 않으나, **엔티티 이름과 필드 명칭**은 자바 클래스와 동일하게 대소문자를 엄격히 구분해야 합니다.

**4. 사용 예시**
```
// Member 엔티티 객체를 대상으로 조회 (테이블 명이 아닌 클래스 명 사용)
String jpql = "select m from Member m where m.name = :username";

List<Member> result = em.createQuery(jpql, Member.class)
    .setParameter("username", "kim")
    .getResultList();
```
### Fetch Join란?
JPQL에서 성능 최적화를 위해 제공하는 기능으로, 연관된 엔티티나 컬렉션을 **한 번의 SQL 쿼리로 함께 조회**하는 방법입니다. 

**1. 정의 및 목적**

- **정의:** SQL의 조인 종류가 아닌, JPQL에서 성능 최적화를 위해 사용하는 전용 문법입니다.
- **목적:** 지연 로딩(LAZY) 설정으로 인해 발생하는 **N+1 문제**를 해결하기 위해, 처음부터 필요한 연관 데이터를 한꺼번에 메모리에 올리는 것입니다.

**2. 핵심 특징**

- **객체 그래프 유지:** 조회 시점에 연관된 엔티티가 프록시(Proxy)가 아닌 **실제 엔티티**로 채워집니다. 따라서 영속성 컨텍스트를 벗어나도 데이터를 사용할 수 있습니다.
- **즉시 로딩과의 차이:** 글로벌 패치 전략(EAGER)은 애플리케이션 전체에 영향을 주지만, Fetch Join은 **해당 쿼리를 실행하는 시점**에만 즉시 로딩이 동작하도록 명시적으로 제어합니다.
- **문법:** `JOIN FETCH` 키워드를 사용합니다.
    - 예: `SELECT m FROM Member m JOIN FETCH m.team`

**3. 기술적 제약 및 주의사항**

- **별칭(Alias) 사용 불가:** 기본적으로 Fetch Join 대상에는 별칭을 줄 수 없습니다. (Hibernate는 지원하지만, 데이터 일관성 문제로 권장되지 않음)
- **둘 이상의 컬렉션 Fetch Join 불가:** 두 개 이상의 컬렉션(`OneToMany`)을 동시에 Fetch Join하면 **카테시안 곱(Cartesian Product)**이 발생하여 데이터 정합성이 깨지거나 성능이 급격히 저하될 수 있습니다. (MultipleBagFetchException 발생 원인)
- **페이징(Paging) API 사용 제한:**
    - **일대다(Collection) 관계**를 Fetch Join하면서 페이징 처리를 하면, Hibernate는 모든 데이터를 메모리에 읽어온 뒤 메모리에서 페이징을 처리합니다. 이는 심각한 **OutOfMemoryError**를 유발할 수 있습니다.
    - 단일 객체(`ManyToOne`, `OneToOne`) 관계는 페이징 처리가 가능합니다.

**4. 해결 효과**

- **쿼리 횟수 감소:** N+1문제에서 발생하는 N번 발생할 추가 쿼리를 1번의 조인 쿼리로 줄여 네트워크 비용을 절감합니다.
### @EntityGraph란?
엔티티를 조회할 때 연관된 엔티티를 **한꺼번에 조회(Fetch Join)**하기 위해 사용하는 어노테이션입니다.

**1. 정의 및 목적**

- **정의:** JPQL 작성 없이도 쿼리 메소드나 JPQL에 연관 관계를 지정하여 특정 엔티티 그래프를 함께 로딩하도록 설정하는 기능입니다.
- **목적:** 기본적으로 지연 로딩(LAZY)으로 설정된 연관 관계를 특정 상황에서만 **즉시 로딩(EAGER)**처럼 동작하게 하여 **N+1 문제**를 해결합니다.

**2. 주요 특징**

- **Fetch Join의 선언적 대안:** JPQL에 직접 `JOIN FETCH` 문법을 쓰지 않고, 어노테이션 설정을 통해 객체 그래프를 지정할 수 있습니다.
- **유연성:** 같은 엔티티 조회 쿼리라도 메서드에 따라 서로 다른 연관 관계 데이터를 함께 가져오도록 선택적으로 적용 가능합니다.
- **로딩 방식 (Fetch Type):**
    - **Fetch:** 설정한 속성은 EAGER로, 나머지는 LAZY로 로딩합니다. (가장 많이 사용됨)
    - **Load:** 설정한 속성은 EAGER로, 나머지는 엔티티에 정의된 기본 전략을 따릅니다.

**3. 사용 방식**

- **NamedEntityGraph:** 엔티티 클래스에 미리 이름을 붙여 정의해두고 재사용하는 방식입니다.
- **Ad-hoc EntityGraph (Spring Data JPA):** 리포지토리 메서드 위에 `@EntityGraph`를 선언하고 `attributePaths` 속성에 연관 필드명을 기입하는 방식입니다.

**4. Fetch Join과의 차이점**

- **문법:** Fetch Join은 JPQL 내부에 명시하며, @EntityGraph는 메서드 설정(어노테이션) 방식입니다.
- **조인 방식:** Fetch Join은 기본적으로 `INNER JOIN`을 사용하지만, @EntityGraph는 연관 관계가 선택적일 수 있어 `LEFT OUTER JOIN`을 기본으로 사용합니다.

**5. 주의사항 및 제약**

- **복잡성:** 아주 복잡한 객체 그래프를 조회할 때는 JPQL을 직접 쓰는 것보다 가독성이 떨어질 수 있습니다.
- **중복 데이터:** Fetch Join과 마찬가지로 일대다(Collection) 관계 조회 시 카테시안 곱(Cartesian Product) 현상이 발생할 수 있으므로 주의해야 합니다.
### commit과 flush 차이점은?
**1. 핵심 정의**

- **Flush:** 영속성 컨텍스트의 **변경 내용을 데이터베이스에 동기화**하는 작업입니다. (쓰기 지연 SQL 저장소의 쿼리를 DB로 전송)
- **Commit:** 데이터베이스의 **트랜잭션을 확정** 짓는 작업입니다. (변경 사항을 영구적으로 저장하고 트랜잭션 종료)

**2. 상세 차이점 비교**

| **구분**      | **flush**      | **Commit**               |
|-------------|----------------|--------------------------|
| **주요 역할**   | 영속성 컨텍스트  DB 동기화 | DB 트랜잭션 확정 및 종료        |
| **영향 범위**   | DB에 쿼리만 전달 (반영 전)       | DB에 실제 데이터 반영 및 잠금 해제 |
| **수행 후 상태** | 트랜잭션 유지 (롤백 가능)    | 트랜잭션 종료 (롤백 불가)         |
| **발생 시점**   | 명시적 호출, 트랜잭션 커밋 전, 쿼리 실행 전   | 비즈니스 로직 종료 시점         |

**3. 기술적 동작 특성**

- **Flush의 동작:**
    1. **변경 감지(Dirty Checking):** 영속성 컨텍스트에 보관된 엔티티의 스냅샷과 현재 상태를 비교합니다.
    2. **쿼리 생성:** 수정된 엔티티가 있으면 수정 쿼리를 만들어 **쓰기 지연 SQL 저장소**에 등록합니다.
    3. **전송:** 저장소의 쿼리(INSERT, UPDATE, DELETE 등)를 데이터베이스에 전송합니다.
    - *주의:* Flush가 일어나도 DB에만 쿼리가 전송된 상태일 뿐, 다른 트랜잭션에서는 이 변경 사항을 볼 수 없습니다(격리 수준에 따라 상이).
- **Commit의 동작:**
    - JPA는 트랜잭션을 커밋할 때 **자동으로 Flush를 수행**합니다. 즉, `Commit = Flush + DB Commit`의 과정을 거칩니다.
    - 커밋이 성공하면 데이터베이스의 데이터가 물리적으로 업데이트되며, 작업 내용을 되돌릴 수 없게 됩니다.

**4. 영속성 컨텍스트와의 관계**

- Flush는 **영속성 컨텍스트를 비우지 않습니다.** 컨텍스트 내의 엔티티 상태를 유지하면서 DB와 싱크만 맞추는 것입니다.
- 영속성 컨텍스트의 내용을 비우는 것은 `em.clear()` 또는 트랜잭션 종료 후 컨텍스트 자체가 소멸될 때 일어납니다.
