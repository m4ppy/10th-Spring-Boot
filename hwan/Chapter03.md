#### 프레임워크 vs API vs 라이브러리의 차이
**프레임워크 : 개발자들이 애플리케이션을 개발하는데 사용되는 구조를 제공하고, 개발자가 개발 시, 이러한 규칙과 구조를 따르도록 하는 것입니다. (주도권은 프레임워크한테 있음)**  
ex) 기본적인 인프라가 공사가 끝난 뼈대 건물로, 화장실 1층 배관을 오른쪽에 뚫었으니까 변기(코드)는 무조건 오른쪽 방(특정 폴더)에 설치해야 합니다. 안그러면 물이 내려가지 않습니다.

**API : 서로 다른  컴포넌트들이 데이터를 주고 받을 수 있도록 만들어진 약속 혹은 규칙**  
ex) 식당에 갔을 때 메뉴판과 점원을 API라고 할 경우, 손님은 식당 내부를 알 필요 없이 메뉴판을 보고 점원(API)에게 주문하면 점원이 주방에 전달하여 요리(Response)를 가져다 줍니다.

**라이브러리 :특정 기능들을 쓰기 편하게 미리 만들어둔 코드뭉치입니다. (주도권이 나에게 있음)**  
ex) 필요할 때 내가 주도적으로 꺼내서 사용합니다. 요리 할 때 칼이 필요하면 칼을 꺼내서 쓰고, 국자가 필요하면 국자를 꺼내서 쓰듯이 필요 할 때 호출해서 사용 할 수 있습니다.

#### 프레임워크 vs 라이브러리 비교
프레임워크는 주도권이 프레임워크한테 말했던 예시처럼 제공된 규칙을 따라야 합니다 (기본 공사가 끝난 건물의 배관)
하지만 라이브러리는 주도권이 나에게 있으므로 필요할 때마다 꺼내서 쓰면 됩니다 (요리 할 때의 요리도구 : 칼, 국자 등등)

#### 강한 결합 vs 약한결합
강한 결합은 클래스가 다른 구체적인 클래스에 직접 의존하는 상태로, 한 클래스 내부의 코드가 변경되면 다른 클래스도 같이 수정해야 합니다.
약한 결합은 측정 클래스가 다른 클래스의 추상화된 규격에 의존하는 상태로
ex) 강한 결합은 일체형 충전기로 아이폰(8핀)을 사용하다가 C타입으로 바꾸면 어댑터까지 바꿔야 하지만 약한 결합은 분리형 충전기로 어댑터에는 포트만 뚫려 있고 선만 뻈다 꼈다 할 수 있습니다.

### 2. 핵심 키워드 정리
SOLID 원칙이란?
SRP (단일 책임의 원칙) : 하나의 클래스는 하나의 책임(기능)만 가져야 한다.
OCP (개방 폐쇠 원칙) : 클래스는 확장에 열려있어야 하며, 수정에는 닫혀있어야 한다.
LSP (리스코프 치환 원칙) : 서브타입은 언제나 부모 타입으로 교체할 수 있어야 한다.
ISP (인터페이스 분리 원칙) : 인터페이스는 잘 분리해야 한다.
DIP (의존성 역전 원칙) : 상위모듈이 하위모듈에 의존하지 않고, 하위 모듈이 상위 모듈에 의존해야 한다.

#### DI란?
- 개념 : 객체가 자신이 필요로 하는 부품(의존성)을 스스로 생성(new)하지 않고, 외부에서 만들어진 것을 전달(주입)받아 사용하는 방식입니다. IoC(제어의 역전)를 실현하는 대표적인 디자인 패턴입니다.
- 비유 : 컴퓨터를 맞출 때, 직접 CPU, RAM, 메인보드 같은 부품을 만들어서 조립하는게 아닌, 이미 완성된 부품들을 가져와서 본체에 끼워 넣기만(주입) 하는 것과 같은 원리입니다.
- 의존성(Dependency)이란? : 의존 대상 B가 변하면 그것이 A에 영향을 미친다. (토비의 스프링)

#### IoC란?  
- 프로그램의 전체 주도권(객체생성, 생명주기관리, 메서드 호출 등)이 개발자에서 외부 시스탬(프레임워크, 컨테이너 등)에 넘어가는 것입니다. 개발자는 필요한 부품만 정의하고, 그것들이 언제 어떻게 사용될 지는 프레임워크가 제어합니다.
- 비유 : 운전은 운전자가 직접 운전을 하여 시동, 주행, 정차, 주차 등의 프로세스를 운전자가 직접 통제하지만 자율주행을 하면 운전자는 목적지(필요한 요소)만 설정하고 운전대에서 손을 뗍니다. 이후 차가 알아서 주행 흐름을 제어하고 목적지까지 데려다주는 것입니다.

#### 생성자 주입 vs 수정자, 필드 주입 차이는?

- 생성자 주입은 객체를 만드는 순간 의존성이 들어오고, 수정자, 필드 주입은 일단 객체를 만든 후 의존성을 넣습니다.
- 생성자 주입은 의존성을 바꿀 수 없지만, 수정자/필드 주입은 의존성 교체가 가능합니다
- 생성자 주입은 의존성 없이 객체가 생성되지 않고 컴파일 단계에서 막히지만, 수정자/필드 주입은 의존성 없이도 객체가 만들어지지만 나중에 해당 기능을 쓸 때 **NullPointerException**이 터집니다
- 생성자 주입은 순환 참조가 실행 시점에 감지되어 실행되지 않지만, 수정자/필드 주입은 실행은 되지만 기능을 호출하는 순간 스택오버플로우가 터집니다.
- 테스트 할 때에 생성자 주입은 자바 코드로 Mock 데이터를 넣을 수 있지만 필드 주입은 스프링 컨테이너를 직접 띄워야 Mock을 넣을 수 있기 때문에 테스트가 무거워집니다.


| 구분 | 생성자 주입 | Setter(수정자) 주입 | 필드 주입 |
|------|------------|-------------------|---------|
| **주입 시점** | 객체 생성 시 | 객체 생성 후 런타임 | 객체 생성 후 런타임 |
| **불변성** | O - `final` 선언 가능, 의존성 변경 불가 | X - 언제든 재주입 가능 | X - 언제든 재주입 가능 |
| **필수 의존성 강제** | O - 주입 없으면 객체 생성 자체가 불가 | X - 주입 없어도 객체 생성됨, NPE 위험 | X - 주입 없어도 객체 생성됨, NPE 위험 |
| **순환 참조 감지** | O - 컴파일/앱 시작 시 즉시 감지 | X - 런타임에 터짐 | X - 런타임에 터짐 |
| **테스트 용이성** | O - 생성자로 Mock 객체 직접 주입 가능 | O - Setter로 Mock 교체 가능 | X - 스프링 컨테이너 없이 테스트 불가 |
| **코드 가독성** | 의존성이 명확히 드러남 | 의존성이 드러남 | 코드는 짧지만 의존성 파악 어려움 |


#### AOP란?
- 관점 지향 프로그래밍 (Aspect Oriented Programming) : 반복되고 공통적으로 사용되는 부분을 분리함으로써 모듈성을 증가시키는 프로그래밍 방법입니다.

- **Aspect** : 공통 기능(로그, 보안 등)을 하나로 묶은 모듈. Pointcut + Advice를 합친 개념입니다.
- **Join Point** : 부가 기능을 끼워넣을 수 있는 모든 시점. 스프링에서는 메서드 호출 시점이 곧 Join Point입니다.
- **Advice** : Join Point에서 실제로 실행되는 부가 기능 코드 그 자체입니다.
- **Pointcut** : 수많은 Join Point 중에서 Advice를 실제로 적용할 위치를 골라내는 조건입니다.
- **Target** : 부가 기능이 적용되는 대상, 즉 핵심 비즈니스 로직을 가진 객체입니다.
- **Advisor** : 스프링 AOP에서만 쓰는 용어로, Advice 1개 + Pointcut 1개를 묶은 가장 작은 단위의 Aspect입니다.


### AOP VS 모듈화의 차이
모듈화 : 핵심 기능을 각 기능별로 코드를 구현 (로그인, 회원가입 등)
AOP : 반복되는 부가 작업 코드를 모듈화해서 공통으로 구현 (에러처리, 보안검사, 로그기록 등)


#### 서블릿이란?
- 개념 : 동적 웹 페이지를 만들 때 사용되는 자바 기반의 웹 애플리케이션 프로그래밍 기술, 웹 상의 통신에서 다양한 요청과 응답의 규칙을 개발자가 처리하는 게 아닌, 비즈니스 로직에만 집중 할 수 있도록 이러한 웹 요청과 응답의 흐름을 간단한 메서드 호출만으로 체계적으로 다룰 수 있게 해주는 기술입니다.
- 서블릿의 동작
![Servlet](Servlet.png)
1. **클라이언트가 HTTP 요청을 보냅니다.**
   브라우저 주소창에 URL을 입력하거나 버튼을 클릭하는 등의 행위로 HTTP Request가 발생합니다. 이 요청은 네트워크를 통해 서버의 **Servlet Container**(ex. Tomcat)에 도착합니다.

2. **Servlet Container가 요청/응답 객체를 생성합니다.**
   Servlet Container는 요청을 받는 즉시 두 가지 객체를 만듭니다.
    - `HttpServletRequest` : 클라이언트가 보낸 요청 정보(URL, 파라미터, 헤더, 쿠키 등)를 담는 객체
    - `HttpServletResponse` : 서버가 클라이언트에게 돌려줄 응답을 담을 빈 객체
      이후 모든 처리는 이 두 객체를 통해 이루어집니다.

3. **web.xml로 어떤 Servlet이 처리할지 결정합니다.**
   Servlet Container는 `web.xml`(또는 `@WebServlet` 어노테이션)에 정의된 URL 매핑 규칙을 보고,
   요청된 URL을 처리할 수 있는 Servlet이 무엇인지 찾습니다.
   ex) `/login` 요청 → `LoginServlet`이 담당하도록 매핑되어 있으면 해당 Servlet으로 연결합니다.

4. **service() → doGet() / doPost()가 순서대로 호출됩니다.**
   Servlet Container가 해당 Servlet의 `service()` 메소드를 호출합니다.
   `service()`는 요청의 HTTP 메소드(GET / POST 등)를 확인한 뒤, 자동으로 적절한 메소드에 위임합니다.
    - 브라우저 주소창 입력, 링크 클릭 → `doGet()` 호출
    - 로그인 폼 제출, 데이터 전송 → `doPost()` 호출

5. **doGet() / doPost()에서 응답을 만들어 돌려줍니다.**
   개발자가 작성한 비즈니스 로직(DB 조회, 연산 등)이 실행되고, 그 결과를 바탕으로 동적 HTML 페이지(또는 JSON 등)를 생성합니다.
   완성된 응답은 `HttpServletResponse` 객체에 담겨 클라이언트에게 전달됩니다.

6. **요청/응답 객체를 소멸시킵니다.**
   응답이 완료되면 Servlet Container가 2번에서 만든 `HttpServletRequest`와 `HttpServletResponse` 두 객체를 메모리에서 제거합니다.
   Servlet 자체는 메모리에 유지되어 다음 요청에도 재사용되지만, 요청마다 생성된 이 두 객체는 해당 요청이 끝나면 사라집니다.

#### 서블릿 vs 서블릿 컨테이너 vs 디스패처 서블릿
서블릿 : 클라이언트의 요청을 처리하고, 동적인 결과를 만들어내는 자바 웹 프로그래밍 기술입니다.
서블릿 컨테이너 : 수많은 서블릿들의 생성, 초기화, 실행, 종료 등의 생명주기를 관리하고, HTTP 요청을 받아 적절한 서블릿에 전달한 뒤 그 결과를 응답으로 반환하는 실행 환경입니다. 대표적으로 톰캣(Tomcat)이 있습니다.
디스패처 서블릿 : 스프링 MVC에서 클라이언트 요청을 공통으로 받아 적절한 컨트롤러에 전달하는 프론트 컨트롤러 서블릿입니다.

### 3. 미션 : 스프링의 전체적인 요청, 응답 흐름 설명하기
### 3. 미션 : 스프링의 전체적인 요청, 응답 흐름 설명하기
```
1. 클라이언트 요청 (Client Request)
- 클라이언트가 특정 URL로 HTTP 요청을 보냅니다.

2. 디스패처 서블릿 (DispatcherServlet)
- 모든 요청은 디스패처 서블릿에 제일 먼저 도착하여 처리됩니다. 디스패처 서블릿은 web.xml 혹은 스프링부트의 자동 설정에 의해 설정됩니다.
- 추가로 설명하자면, 서블렛 3.x 이전에는 Spring MVC 애플리케이션의 web.xml 설정 파일에 등록하여 어떤 디스패처 서블렛을 사용해야 하는지 서블릿 컨테이너에 알려줘야 했습니다. 하지만, 3.x 이후 버전부터는 ServletContainerInitializer를 사용하여 자바 코드만으로 서블릿을 등록 할 수 있게 되었습니다. 그리고 현재 Spring Boot 도입 이후에는 Spring-boot-starter-web 라이브러리만 추가하면 스프링 부트의 내장된 톰캣이 서블릿 등록 과정을 자동으로 처리해줍니다.

3. 핸들러 매핑 (Handler Mapping)
- 디스패처 서블렛은 요청 URL 기반으로 적절한 컨트롤러를 찾기 위해 여러 핸들러 매핑 전략을 사용하여 다양한 핸들러를 매핑합니다.

4. 핸들러 어댑터 (Handler Adapter)
- 디스패처 서블릿은 핸들러 매핑이 찾아준 컨트롤러를 실행 할 수 있는 적절한 핸들러 어댑터를 선택합니다.
- 핸들러 어댑터는 각 요청을 처리하는 실제 컨트롤러의 메소드를 호출하고 결과를 반환합니다.

5. 컨트롤러 (Controller)
- 컨트롤러는 클라이언트의 요청을 처리하고, 모델 데이터를 준비하여 논리적 뷰로 반환합니다.

6. 모델 (Model)
- 모델은 컨트롤러에 의해 준비된 데이터를 뷰에 전달합니다

7. 뷰 리졸버 (View Resolver)
- 컨트롤러가 반환한 논리적 뷰 이름을 실제 뷰 객체로 변환하기 위해 디스패처 서블릿은 뷰 리졸버를 사용합니다.
- DTO(Data Transfer Object)를 사용하여 ObjectMapper를 통해 JSON 응답으로 보내는 경우에는 뷰 리졸버 과정은 생략됩니다.

8. 뷰 (View)
```
